# AGENTS.md

## Project Overview

Legacies is a command-line autonomous simulation game.

Core philosophy:

- Simulation first, game second.
- Performance matters.
- The SimulationEngine is the backbone of the project.
- World Generation and Active Play both run on the same simulation engine.
- Cause-and-effect is non-negotiable.
- Everything meaningful in the world should be explainable by prior state and prior events.
- The Chronicle is the primary player-facing output.

The player does not control an abstract omniscient empire game. The player eventually selects a real polity/society generated by the simulation and experiences the world through that lens.

---

## Non-Negotiable Design Rules

### 1. Cause and Effect
Every important world outcome must come from prior simulation state.

Do not introduce:
- unexplained jumps in state
- hidden magical promotions
- arbitrary narrative-only outcomes
- systems that bypass established world logic

If something changes, the code should make it possible to answer:
- what changed
- why it changed
- which prior conditions caused it

### 2. Shared Simulation Backbone
World Generation and Active Play must use the same SimulationEngine and the same core world rules.

Do not build one-off fake worldgen logic that creates outcomes impossible in live simulation unless there is a very strong reason and it is explicitly documented.

### 3. Performance First
Legacies is a simulation-heavy project. Avoid wasteful abstractions, unnecessary allocations, and overly chatty per-tick work.

Prefer:
- simple data flow
- clear ownership
- deterministic processing
- profiling-friendly structures
- batching where it helps

### 4. Chronicle Is the Player Experience
The Chronicle is not debug spam. It is the main player-facing history stream.

Chronicle messages should be:
- readable
- grounded in simulation truth
- meaningful
- not redundant
- not flooded by low-value noise

### 5. Honest World State
The game must not fabricate strength, maturity, or stability for convenience.

If the world is weak, sparse, unstable, or uneven, surface that honestly.

## Code Structure Guidance

### Legacies.Domain
Contains the simulation model and core game/domain logic.

Expected long-term responsibilities:
- world state
- entities and value objects
- simulation systems
- world generation logic
- discoveries / advancements
- polity and society logic
- chronicle event generation rules

### Legacies.Client
Contains the command-line entrypoint and player-facing console experience.

Expected responsibilities:
- startup
- run configuration
- loading/saving bootstrap
- screen/view routing
- player input handling
- rendering the chronicle and world views

Do not let the client become the home of simulation rules.

---

## Naming Guidance

Use `SimulationEngine` as the canonical name for the main monthly simulation runner.

Prefer explicit, domain-meaningful names over generic names like:
- `Manager`
- `Helper`
- `Util`
- `Stuff`
- `Processor`

Good names describe responsibility and world meaning.

---

## Implementation Style

Prefer:
- straightforward code over clever code
- explicit data flow over hidden side effects
- deterministic behavior over incidental behavior
- small composable systems with clear ownership
- domain language that matches the project docs

Avoid:
- premature micro-abstractions
- speculative framework-building
- giant god classes
- debug-only terms leaking into player-facing systems

---

## Simulation Rules for Early Development

During early development, prioritize this order:

1. world representation
2. monthly simulation loop
3. ecology / support / survival basics
4. species and population continuity
5. movement and settlement logic
6. society/polity emergence
7. chronicle output
8. world generation flow
9. player-facing selection and active-play handoff

Do not jump to late-game complexity before the backbone is stable.

---

## Testing and Validation Philosophy

When adding or changing simulation systems, validate:

- determinism under fixed seeds
- cause-and-effect traceability
- no impossible state transitions
- no chronicle spam / duplicate chronicle messages
- no obvious performance regressions

For simulation features, correctness and explainability matter as much as raw output.
