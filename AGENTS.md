# AGENTS.md

## Project Overview

Legacies is a command-line autonomous simulation game.

Core philosophy:

- Simulation first, game second.
- Performance matters.
- The SimulationEngine is the backbone of the project.
- World Generation and Active Play both run on the same simulation engine.
- Cause-and-effect is non-negotiable.
- Everything meaningful in the world should be explainable by prior state and prior events.
- The Chronicle is the primary player-facing output.

The player does not control an abstract omniscient empire game. The player eventually selects a real polity or society generated by the simulation and experiences the world through that lens.

---

## Non-Negotiable Design Rules

### 1. Cause and Effect

Every important world outcome must come from prior simulation state.

Do not introduce:
- unexplained jumps in state
- hidden magical promotions
- arbitrary narrative-only outcomes
- systems that bypass established world logic

If something changes, the code should make it possible to answer:
- what changed
- why it changed
- which prior conditions caused it

### 2. Shared Simulation Backbone

World Generation and Active Play must use the same SimulationEngine and the same core world rules.

Do not build one-off fake world generation logic that creates outcomes impossible in live simulation unless there is a very strong reason and it is explicitly documented.

### 3. Pressure-First Simulation

Legacies uses a pressure-first simulation model.

Core rule:
- important simulation decisions should be driven by pressures, not isolated one-off event logic

Examples:
- populations should move because movement pressure rose
- societies should fragment because cohesion and strain pressures worsened
- settlements should be abandoned because viability and abandonment pressures crossed meaningful thresholds

Preferred causal pattern:
1. conditions or events change world state
2. those changes modify one or more pressures
3. later systems evaluate the updated pressure state
4. behavior and outcomes emerge from those pressures

This preserves:
- cause-and-effect
- explainability
- determinism
- consistency across world generation and active play

#### Pressure Ownership

Each major entity type should own the pressures relevant to its layer of simulation.

Examples:
- regions own environmental and ecological pressures
- populations own survival, movement, and recovery pressures
- societies own cohesion, continuity, and fragmentation pressures
- settlements own viability, growth, and abandonment pressures
- polities own legitimacy, stability, and structural pressures

Use a shared pressure framework, but keep pressure ownership entity-specific.

#### System Responsibility

Broad systems should primarily write the pressure state for their own layer.

Examples:
- `EnvironmentSystem` writes environmental pressures
- `EcologySystem` writes support and scarcity pressures
- `PopulationSystem` writes biological and survival-state response
- `MovementSystem` writes movement, displacement, and local opportunity pressures
- `SocialSystem` writes cohesion and fragmentation pressures
- `SettlementSystem` writes site-viability pressures
- `PoliticalSystem` writes legitimacy and structural pressures

Systems may read upstream pressures, but they should mainly own their own pressure outputs.

#### Events and Chronicle

Events should usually modify pressures rather than bypass causality and force downstream outcomes directly.

Chronicle output should surface the meaningful consequences of pressure-driven change, not raw pressure telemetry.

#### Geography Rule

Regions are pure regions and are the canonical spatial unit of simulation.

Do not create a hidden or hybrid cell-based model that becomes the real simulation map underneath the region model.

Allowed:
- region adjacency
- region traits
- region descriptors
- route links between regions

Not desired:
- a second true spatial layer that competes with regions as world truth

The simulation should think in regions, because the player understands the world through regions.

#### Documentation Rule

`PRESSURES.md` is canonical guidance for the pressure model.

When pressure semantics, ownership, or propagation rules change, update `PRESSURES.md` and any affected canonical design docs in the same work.

### 4. Performance First

Legacies is a simulation-heavy project. Avoid wasteful abstractions, unnecessary allocations, and overly chatty per-tick work.

Prefer:
- simple data flow
- clear ownership
- deterministic processing
- profiling-friendly structures
- batching where it helps

### 5. Chronicle Is the Player Experience

The Chronicle is not debug spam. It is the main player-facing history stream.

Chronicle messages should be:
- readable
- grounded in simulation truth
- meaningful
- not redundant
- not flooded by low-value noise

### 6. Honest World State

The game must not fabricate strength, maturity, or stability for convenience.

If the world is weak, sparse, unstable, or uneven, surface that honestly.

## Code Structure Guidance

### Legacies.Domain

Contains the simulation model and core game or domain logic.

Expected long-term responsibilities:
- world state
- entities and value objects
- simulation systems
- world generation logic
- discoveries and advancements
- polity and society logic
- chronicle event generation rules
- pressure ownership and pressure propagation logic

### Legacies.Client

Contains the command-line entrypoint and player-facing console experience.

Expected responsibilities:
- startup
- run configuration
- loading and saving bootstrap
- screen and view routing
- player input handling
- rendering the chronicle and world views
- presentation pacing

Do not let the client become the home of simulation rules.

## Naming Guidance

Use `SimulationEngine` as the canonical name for the main monthly simulation runner.

Prefer explicit, domain-meaningful names over generic names like:
- `Manager`
- `Helper`
- `Util`
- `Stuff`
- `Processor`

Good names describe responsibility and world meaning.

Broad systems are preferred early where they preserve clear ownership boundaries.

Examples:
- `EnvironmentSystem`
- `EcologySystem`
- `PopulationSystem`
- `MovementSystem`
- `SocialSystem`
- `SettlementSystem`
- `KnowledgeSystem`
- `InteractionSystem`
- `PoliticalSystem`
- `ChronicleSystem`
- `EvaluationSystem`

Avoid splitting architecture into excessive micro-systems too early unless there is a strong need.

---

## Implementation Style

Prefer:
- straightforward code over clever code
- explicit data flow over hidden side effects
- deterministic behavior over incidental behavior
- small composable domain models with clear ownership
- broad systems with coherent responsibilities
- domain language that matches the project docs

Avoid:
- premature micro-abstractions
- speculative framework-building
- giant god classes
- debug-only terms leaking into player-facing systems
- one-off event logic that bypasses the pressure model

---

## Simulation Rules for Early Development

During early development, prioritize this order:

1. world representation
2. monthly simulation loop
3. environment and ecology foundation
4. species and population continuity
5. movement and settlement logic
6. society and polity emergence
7. chronicle output
8. world generation flow
9. player-facing selection and active-play handoff

Do not jump to late-game complexity before the backbone is stable.

## Broad System Order

The intended broad monthly order is:

1. `CalendarSystem`
2. `EnvironmentSystem`
3. `EcologySystem`
4. `PopulationSystem`
5. `MovementSystem`
6. `SocialSystem`
7. `SettlementSystem`
8. `KnowledgeSystem`
9. `InteractionSystem`
10. `PoliticalSystem`
11. `ChronicleSystem`
12. `EvaluationSystem`

This order exists to preserve causality.

General flow:
- time context updates first
- environmental and ecological pressures are established
- populations respond biologically
- movement responds spatially
- social and settlement consequences follow
- knowledge, interaction, and politics build on the settled month
- chronicle surfaces meaningful outcomes
- evaluation observes the final monthly state without rewriting it

---

## Geography Guidance

Regions are the core spatial simulation unit.

Use:
- region adjacency
- region traits
- region-level ecology
- region-connected routes
- region-scoped settlement and population state

Avoid:
- hidden cell simulation layers that become the real geography
- presentation-only regions wrapped around some deeper true map model

If the player experiences the world through regions, the simulation should also operate primarily through regions.

---

## Pressure Guidance

Pressure semantics should remain clear and explicit.

When implementing or changing a pressure-driven system:

- define which entity owns the pressure
- define which system writes it
- define which downstream systems consume it
- define how it accumulates, decays, or is relieved
- define what sources contribute to it when practical

Prefer source-aware pressures where feasible so the game can explain:
- why a pressure rose
- what factors contributed to it
- why an entity changed behavior

Do not create:
- giant undifferentiated pressure blobs
- incompatible per-system pressure logic
- direct event-to-outcome shortcuts that bypass causal pressure flow

---

## Testing and Validation Philosophy

When adding or changing simulation systems, validate:

- determinism under fixed seeds
- cause-and-effect traceability
- pressure-driven behavior remains explainable
- no impossible state transitions
- no chronicle spam or duplicate chronicle messages
- no obvious performance regressions

For simulation features, correctness and explainability matter as much as raw output.

## Working Rule for Contributors and Agents

Before starting major work:

1. Read the canonical docs.
2. Confirm the intended pressure ownership and causal flow.
3. Confirm whether the work belongs in the domain or the client.
4. Implement the smallest truthful version first.
5. Sync docs before considering the work complete.

When a design decision changes:
- update the relevant canonical docs in the same work
- especially update `PRESSURES.md` when pressure semantics change
- update `IMPLEMENTED_SYSTEMS_LIST.md` when implementation status changes
